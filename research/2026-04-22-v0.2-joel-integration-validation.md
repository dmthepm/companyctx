# v0.2 Joel-integration validation — 119 sites, 30 head-to-heads, 55 edge cases

## 1. Summary — verdict

**Not yet Joel-ready for a drop-in Phase 1 swap, but closer than the
`v0.2.0` release-notes headline reads, and the gap is legible.** On a
119-site run stratified across all 17 of @joel-req's live D100 niches,
`companyctx fetch` returned envelope `status: "ok"` on **116 / 119
(97.5 %)** — matching the v0.1 durability headline on a broader,
medical-and-home-services-diverse sample. Zero sites hit
`blocked_by_antibot`; the Chrome 146 TLS impersonation is carrying.

The gap is not in the fetch rate — it is in **what the `ok` envelopes
actually contain**. **31 / 119 (26.1 %)** of `status: "ok"` runs came
back with **under 1 KiB of extracted `pages.homepage_text`** (FM-7
thin-body). That's the exact silent-success pattern
[#79 (COX-44)](https://github.com/dmthepm/companyctx/issues/79) calls
out, and it is disproportionately concentrated in the niches where Joel
is shipping briefs today (real-estate photography, virtual staging,
gutter-installation-repair — see §3). Those 31 `ok`-but-thin envelopes
are technically partial data wearing an `ok` label; downstream synthesis
has no way to branch on them without a byte-length check the SPEC
doesn't currently require.

On the 30 head-to-head pairs (companyctx output vs partner's already-
rendered `research-brief.md` for the same host), the raw-text muscle
holds up on the buckets it is designed to cover: **differentiator**,
**audience**, and **credentials** were coverable from raw `pages_text`
on 26 / 30, 26 / 30, and 25 / 30 respectively — via_LLM or covered_now.
**Reviews** (29 / 30 out-of-scope) and **media mentions** (30 / 30
out-of-scope) behave exactly as the SPEC boundary predicts — those
buckets need direct-API providers (#7, #58) that v0.2 does not ship.
**Content/social** (28 / 30 out-of-scope) is the honest boundary: handle
*presence* is sometimes derivable from the page text, but follower
*counts* are not in v0.2 scope and should not be expected to be.

On the CLI-surface edge-case pass, the schema-locked envelope and the
structured-error shape hold under unusual inputs, with three concrete
defects worth filing (see §5): `validate` silently accepts envelopes
missing `schema_version` (contradicts SPEC §112); NXDOMAIN routes to
`ssrf_rejected` instead of `no_provider_succeeded`; and
`path_traversal_rejected` only guards `--mock` fixture paths, not URL
paths (the error-code name implies broader coverage than the code
delivers).

**Go/no-go:** **not-yet-go on a Phase 1 swap; go on the v0.3 roadmap
direction.** The shape of what's missing (#79 empty-body detection, #7
Google Places for reviews, #9 cache so re-runs are cheap) is exactly
what the roadmap is already pointing at. Don't pivot v0.3; sharpen the
honesty on FM-7 and ship the Places-API provider.

## 2. Phase 1 coverage — what does the partner's LLM-reads-HTML step actually do?

The canonical contract lives at
[`phase1-research.md`](/Users/devonmeadows/Documents/GitHub/new-signal-studio/.claude/skills/d100/references/phase1-research.md)
in the partner skill repo. The agent runs three steps per prospect —
website fetch, web search, social scan — and emits a fixed brief
template with six named sections. Read against what the v0.2 envelope
populates on `data.pages.*`, the coverage maps cleanly.

The grading rubric used through this report (set once, applied to every
row in §4):

- **covered_now** — the raw value lives directly on `data.pages.*`
  (e.g. `services[]`, a detected `tech_stack` fingerprint, a team-member
  name in homepage prose, a linked-out social handle in `homepage_text`).
- **covered_via_LLM** — the signal is present somewhere in
  `pages.homepage_text` or `pages.about_text` at enough density that an
  LLM synthesis call could re-derive the brief bucket from it without
  another fetch. This is the v0.2 wedge: we hand the synthesis layer
  raw text; it does the phrasing.
- **out_of_scope_in_v0.2** — the bucket requires a provider the
  Deterministic Waterfall does not ship today (Google Places #7, Yelp
  Fusion, YouTube Data, Brave search #58).
- **bug_thin_body** — the site plainly has the source content the brief
  pulls from, but `companyctx` returned `status: "ok"` with thin/empty
  `homepage_text`. This is FM-7 / #79 terrain.

### URL input variance — 21 classes

Every one of the 8 listed classes in the issue was exercised, plus 13
more shapes for completeness. Summary of behaviors — all results are
with the CLI default (`companyctx fetch <input> --json`):

| Input shape | Envelope outcome | Comment |
|---|---|---|
| `example.com` (no scheme) | `ok` | happy path; scheme defaulted |
| `http://example.com` / `https://example.com` | `ok` | both work |
| `HtTp://example.com` (mixed case) | `ok` | scheme case-folded |
| `www.example.com` | `ok` | www prefix tolerated |
| `example.com/` (trailing slash) | `ok` | tolerated |
| `example.com/about` (path) | `ok` | path preserved |
| `example.com:80` | `partial` / `no_provider_succeeded` | port 80 redirects to 443; fetch then fails |
| `example.com:8080` | `partial` / `no_provider_succeeded` | generic failure, not a port-rejection message |
| `example.com/?utm_source=x` | `ok` | query string tolerated |
| `example.com#about` | `ok` | fragment stripped |
| `例.jp` (IDN) | `partial` / `no_provider_succeeded` | degrades quietly; no IDN-specific error surface |
| `93.184.216.34` (IPv4 literal) | **timeout at 20 s** | `rc = -999` — subprocess hit the harness 20 s cap rather than the provider's 10 s |
| `[2606:2800:...]` (IPv6 literal) | `partial` / `no_provider_succeeded` | no IPv6-specific surface |
| `localhost` | `partial` / `ssrf_rejected` | correctly blocked |
| `file:///etc/passwd` | `partial` / `ssrf_rejected` | correctly blocked |
| `gopher://example.com` | `partial` / `ssrf_rejected` | correctly blocked |
| `data:text/html,<h1>x</h1>` | `partial` / `ssrf_rejected` | correctly blocked |
| `example.com/<2100-a-path>` (>2 KB URL) | `ok` | very-long URL tolerated |
| `""` (empty) | `partial` / `ssrf_rejected` | correctly rejected |
| `" "` (whitespace) | `partial` / `ssrf_rejected` | correctly rejected |

**Surprises:**

- **IPv4-literal timeout (20 s)** — `93.184.216.34` is example.com's
  advertised IP. A bare-IP fetch should either resolve fast or be
  rejected. Worth a follow-up issue; for now the SSRF pre-flight's
  DNS-to-public-IP rule may be re-doing work.
- **Port 8080 + IDN + IPv6** all degrade to the same generic
  `no_provider_succeeded` — acceptable, but the error surface does not
  tell an agent *why* they failed in a way that's distinguishable from
  "the site isn't reachable." Would be cheap to expand
  `_classify_error_code` to cover these.

### Piping + stdin

- `fetch … --json | jq '.status'` — works. Clean stdout, no stderr
  noise on the pipe.
- `schema | jsonschema-validate` — works. Schema is externally
  valid JSON Schema Draft 2020-12 (verified with the `jsonschema`
  library on the mock envelope + emitted schema).
- **`echo example.com | companyctx fetch - --json`** — the `-` is
  treated as a literal hostname (`ssrf_rejected`), **not** as a
  stdin-URL pattern. No stdin-URL mode exists. Acceptable for v0.2;
  worth a help-text note.

### Flag combinations + reserved flags

- `--markdown`, `--refresh`, `--from-cache`, `--no-cache`, `--config` —
  all reject with `rc=2` and a tracking-issue pointer (per SPEC §47,
  §60-67). Perfect.
- `--json --markdown` (both) — rejects with `rc=2`. Perfect.
- `--json --json`, `--mock --mock` (repeated) — Typer tolerates both;
  envelope returns normally. Acceptable.
- `--mock` on a host with no fixture → `partial` / `no_provider_succeeded`
  with a reasonable `suggestion`.

### Schema + validate round-trip

- `companyctx schema` emits **11,714 bytes** of valid JSON Schema
  Draft 2020-12.
- External `jsonschema.validate(mock_envelope, schema)` passes.
- `companyctx validate mock-envelope.json` returns `rc=0 ok`.
- **Bug: `companyctx validate` accepts an envelope missing the
  required `schema_version` field.** The Pydantic model defaults
  `schema_version: Literal["0.2.0"] = "0.2.0"`, so a v0.1-shaped
  envelope lacking the field is coerced to v0.2 silently. SPEC §112-113
  explicitly says *"v0.1 envelopes lack the `schema_version` field and
  fail validation under `extra="forbid"`"* — that's the documented
  contract, and the code does not enforce it. **See follow-up issue
  (§9, item A).**
- `companyctx validate empty-file.json` → `rc=1` with a clean error.

### The 7 `error.code` values — what fires, what doesn't

| Code | Triggered via public CLI? | How |
|---|---|---|
| `ssrf_rejected` | **yes — easy** | private IP (10.0.0.1), loopback (127.0.0.1), cloud-metadata IP (169.254.169.254), `file://`, `gopher://`, `data:`, empty / whitespace input |
| `network_timeout` | **no — contrived only** | port 81 on example.com rolls up to `no_provider_succeeded`, not `network_timeout`. Classifier matches on "timeout" substring; `curl_cffi`'s timeout exception does not always carry that word |
| `blocked_by_antibot` | **no on this run** | `nowsecure.nl` (known Cloudflare-challenge site) returned `status: ok` at Chrome 146 fingerprint |
| `path_traversal_rejected` | **no via URL input** | `example.com/../etc/passwd` returns `status: ok` — the code only fires for `--mock` fixture-path escapes (see `companyctx/core.py:421-424`), not for URL paths. The error-code name is broader than its implementation; **see follow-up issue (§9, item C).** |
| `response_too_large` | **no on this run** | hetzner `100MB.bin` returned `no_provider_succeeded` — the provider's `response_too_large:` prefix path fired, but the envelope's classifier rolled up the smart-proxy miss as a generic failure. Unit-testable; not publicly reproducible without a live >10 MiB 200-OK endpoint |
| `no_provider_succeeded` | **yes — easy** | NXDOMAIN, unreachable port, `--mock` with no fixture |
| `misconfigured_provider` | **no in this run** | setting `COMPANYCTX_SMART_PROXY_URL=not-a-valid-url` while Attempt 1 succeeds leaves the envelope `ok`; the smart-proxy provider never runs. The code path is covered in `tests/test_envelope_error_codes.py`; triggering via public CLI alone requires a site Attempt-1 can't reach |

**Net:** 2 / 7 codes fire from a single cold CLI invocation without
contrivance (`ssrf_rejected`, `no_provider_succeeded`). The remaining 5
are real code paths with tests covering them, but they are not
agent-discoverable from CLI-level probing. That's acceptable for an
MIT-licensed Python library that exposes its enum publicly; it is a
documentation-surface gap worth noting.

Third honest finding: **NXDOMAIN routes to `ssrf_rejected`**, not
`no_provider_succeeded`. The SSRF guard rejects "unsafe_url" on DNS
failure; the classifier's first substring hit on `"unsafe_url"` wins
before the NXDOMAIN / resolver path gets a chance. Semantically odd —
an unresolvable domain is not an SSRF attempt. **See follow-up issue
(§9, item B).**

### Providers list — agent parsability

Both `companyctx providers list` (human) and `companyctx providers list
--json` (agent) return cleanly. The JSON form — added in #68 / #80 —
is a list of dicts with the exact shape SPEC §125-144 describes
(`slug`, `tier`, `category`, `cost_hint`, `status`, `reason`). An
agent can branch on `status == "not_configured"` and read `reason`
without any text-parsing. No gap.

### Help text

`--help` on `root`, `fetch`, `schema`, `validate`, `providers`,
`providers list` all return `rc=0` with non-trivial output. The
reserved-flag rejection text on `fetch` points at #9 (cache-related
flags) and #68 (`--markdown`); that's exactly the "agent-actionable"
behavior the acceptance calls for.

### Error-message agent-actionability

Spot-checked the `suggestion` field across 11 non-`ok` envelope
responses (3 from the coverage pass, 8 from the edge-case pass). Every
one carried an `EnvelopeError.suggestion` string that either named the
provider to configure (`"configure a smart-proxy provider key…"`) or
the specific input the caller should change (`"URL rejected: private IP"`).
No empty or generic suggestions observed. Agent-branchability on
`error.code` is fine; the more useful finding here is that agents
**could also branch on `homepage_bytes < 1024`** on `status: ok`
envelopes to detect the FM-7 class. Today they'd need the byte-length
check themselves.

## 6. Time-saved + cost-saved estimate — honesty caveats first

**What we have a data point for:** one committed session trailer —
the 2026-04-21 waste-management-services live run — reports
**`api_cost_usd = 31.19`** across **51 minutes**, **20 prospects**, all
three D100 phases combined. Backfill transcripts don't carry the cost
trailer, so the denominator is n=1 for the Phase-1-replaceable slice.
Treat the numbers below as directional, not benchmarked.

**What `companyctx` actually replaces:** the "fetch homepage(s) and read
the HTML" turn inside Phase 1. Phase 1 per prospect typically does 1-3
website fetches (homepage + maybe an about/team/services page) followed
by an LLM read-through producing a ~500-word research brief.

**Per-prospect token estimate (Sonnet pricing, directional):**

- A typical successful `companyctx fetch` yields ~2 KiB p50 (~500-1000
  tokens) of extracted text. Under the partner's current flow that same
  prospect's WebFetch pulls raw HTML at ~5-30 KiB per page, pre-
  extraction, fed into the LLM as input context. Call it ~10 KiB / 2.5k
  tokens per HTML-reads-turn, 1-3 turns per prospect.
- If the partner's synthesis LLM now reads **2 KiB of cleaned text
  once** instead of **~20 KiB of HTML across 2 turns**, input-token
  savings land around **~5-10× on that specific step** — 3-5k tokens
  saved / prospect on input side. The synthesis output length is
  unchanged; output tokens don't move.
- At Sonnet's $3 / 1M input pricing, that's **~$0.01-0.03 saved per
  prospect per Phase-1 research call**. Across a 20-prospect batch,
  maybe **$0.20-0.60 saved on Phase 1 input tokens** — measurable but
  not load-bearing for the `$31` total batch cost. The real Phase-1
  cost driver is the synthesis + web-search + social-scan turns,
  none of which v0.2 replaces.

**Latency story:** p50 3.97 s / p90 7.87 s means a 20-prospect batch
run purely sequentially through `companyctx` would take ~80 s at p50
floor, ~160 s at p90. In the partner's current flow the WebFetch turn
is LLM-mediated (read-through adds wall-clock); deterministic fetch
takes the cognition step out. **Plausible wall-clock saving: ~5-15 s
per prospect on the Phase-1 read step** for a batch of 20 that runs
subagents in parallel. That is a rounding error next to the 51-minute
batch duration, though — the latency win lives in the per-prospect
agent-call cost model (fewer LLM turns), not the wall-clock total.

**Determinism win:** **this is the honest value prop, and it doesn't
rate-limit on cost.** The partner's transcripts show the agent making
different Phase-1 decisions on different runs against the same
prospect (partial recovery from BBB/Birdeye/Yelp when primary fetches
block, per FM-10 in the register). `companyctx`'s envelope is the
same shape — `status`, `error.code`, `provenance` — regardless of
which Attempt fired. That's the input-to-synthesis contract shape the
partner is missing today, and it is worth more than the token savings.

**Bottom line for §6:** The cost case is real but modest. The case
**for the v0.2 swap is the contract-shape consistency**, not the dollar
savings. **Treat "saves time" as "saves cognition" not "saves tokens"**
and the numbers land honestly.

## 7. What's missing for full Joel-readiness

Three concrete capabilities, ordered by how close they are:

### A. FM-7 empty-body detection + `partial` status — #79 (COX-44)

26.1 % of `ok` envelopes silently under-deliver. That is the single
thing most likely to break a partner swap — downstream synthesis sees
`status: ok`, trusts the envelope, and writes a brief against ~400
bytes of text. The fix lives inside the v0.2 provider contract (bump
status to `partial` + emit a structured error when extracted bytes
fall below a floor) and does not require new schema work. **File-level
blocker for Joel-readiness.**

### B. Google Places direct-API provider — #7 (COX-5)

Reviews (rating + count) is the single biggest out-of-scope bucket in
§4 — 29 / 30 briefs cite a Google rating that `companyctx` leaves
null. Covered in SPEC §71-74 and the scaffolded provider slot
already exists. **Tracking is already there; this validation report is
an argument to ship it, not to re-scope it.**

### C. Cache + batch — #9 (COX-6)

If every `companyctx fetch` is an ~4 s sequential network call, a
20-prospect batch re-running to pick up a fix is ~80 s of wall clock
and 20 × Attempt-1-provider-load. Cache makes the re-run free. `batch
<csv>` is the fan-out wrapper. Both are waterfall-invariant — they
speed up re-runs without changing envelope shape. **Lower urgency than
A + B for the first swap; becomes critical for steady-state use.**

Two smaller gaps worth mentioning but not blocking:

- **Structured-signals provider** (`signals_site_heuristic`, SPEC §71)
  would populate `team_size_claim`, `copyright_year`, `last_blog_post_at`
  without an LLM. None of those are in the brief template today; not
  Joel-blocking, but a cheap win once #79 lands.
- **About-page auto-discovery** (FM-11 in the register). The v0.2
  extractor does not follow an "/about" anchor unless the caller
  passes it. That's why `data.pages.about_text` is often null even
  when `homepage_text` is fat. Not urgent for v0.3; worth an owned
  heuristic if it stays unfixed through v0.4.

## 9. Follow-up issues

Three new issues, all scoped to surface the specific gaps this
validation found. None wider than a single narrow fix.

**A. `companyctx validate` silently accepts envelopes missing
`schema_version`.** SPEC §112-113 states v0.1 envelopes must fail
validation. Today `Envelope.schema_version: Literal["0.2.0"] = "0.2.0"`
has a default and passes. The fix is either to remove the default, or
(better) to add an envelope-root `@model_validator` that asserts the
field was present in the input dict. Low-risk, pure schema-module
change. **Propose: file as a separate issue; link this report; flag
with `docs`/`schema` labels.**

**B. NXDOMAIN routes to `ssrf_rejected`, not `no_provider_succeeded`.**
The SSRF pre-flight raises `"unsafe_url"` on DNS-resolution failure,
and `_classify_error_code` in `companyctx/core.py:419` picks up the
`"unsafe_url"` prefix first. Semantically, NXDOMAIN isn't an SSRF
attempt. Two fixes available: (i) widen the classifier to recognise
DNS-resolver errors explicitly, (ii) split the pre-flight's error
strings so DNS failure and IP-private-range rejection don't share a
prefix. **Propose: file as a separate issue; low severity but
worth a named code-path.**

**C. `path_traversal_rejected` error-code name implies URL-path
protection that the code does not deliver.** Today it only fires for
`--mock` fixture-path escapes
([`companyctx/core.py:421-424`](../companyctx/core.py)). `example.com/../etc/passwd`
is happily fetched. Two options: (i) rename the error code to
`fixture_path_traversal_rejected` (SCHEMA breaking, requires a MINOR
schema_version bump), (ii) expand the code to cover URL-path traversal
too. **Propose: file as a separate issue; tag `schema` label so any
rename decision is visible as a breaking-change surface.**
