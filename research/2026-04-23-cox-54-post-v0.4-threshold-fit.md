# COX-54 — fresh-site threshold-fit on v0.4.0's 1024-byte floor

## Why this is a fresh-live run, not a re-classification

[COX-52 / #91](https://github.com/dmthepm/companyctx/issues/91) re-classified
the archived 209-site COX-46 corpus against the v0.4.0 gate and showed
the honesty contract holds against stored evidence (see
[research/2026-04-23-cox-52-post-fix-reclassification.md](2026-04-23-cox-52-post-fix-reclassification.md)).
That analysis answered "do thin bodies stop hiding as `ok`?" — yes, by
construction.

COX-54 answers a different question: **does the 1024-byte floor
incorrectly catch legitimate thin-but-usable brochure sites?** A one-page
clinic or minimalist services page could fairly return 700-900 bytes of
real text and get classified `degraded + empty_response` — a false
positive the archived data can't disprove because the threshold fires on
a byte count stored in the archive.

This report measures that false-positive rate against **fresh live
traffic** on 30 sites biased toward thin-brochure patterns.

**Primary finding — the floor itself is correctly calibrated.**
Floor-threshold false-positive rate is **0/30 = 0.0 %** on this sample.
Not a single fetch returned 700-1200 bytes of real content that the gate
incorrectly demoted; the two smallest `status: ok` runs (1432 B and
1464 B) both delivered coherent small-business content on LLM-assisted
pre-review (see §LLM-assisted pre-review). The 1024-byte line does not sit in a live distribution of
legitimate thin-but-usable sites.

**Secondary finding (preliminary) — `empty_response` looks like it is
conflating two different failure modes.** Of the 7 runs that landed
`degraded + empty_response` (0 B extracted), an LLM-assisted pre-review
via Claude Code's `WebFetch` tool flagged **6 of 7 as likely
content-bearing sites** (3-20 KB visible to a browser-class fetch),
not thin-brochure emptiness. The exact rate is pending human
confirmation (see §LLM-assisted pre-review).

## Mechanism — why `empty_response` is firing on content-bearing sites

Three hypotheses, each compatible with the observed 0 B:

1. **CDN fingerprint-filter** serves a stripped HTML body to
   `curl_cffi chrome146` impersonation that differs from the one a real
   browser sees. The HTTP status is 200, the body is near-empty; the
   gate correctly reports "empty_response" about *what the provider
   received* but misleadingly implies the site itself is thin.
2. **JS-rendered content shell** — homepage serves a React / Vue / HTMX
   mount point with no crawlable text until hydration. trafilatura sees
   the `<div id="app">` wrapper and nothing else; fetch succeeded,
   there is no content to extract without a real browser.
3. **trafilatura config miss** — content is present in the raw HTML,
   but trafilatura's heuristics (`favor_recall=False`, default
   `min_extracted_size`) filter it out. Less likely given that neither
   `homepage_text` nor `about_text` populated, and the envelopes show
   `provider_status=failed` (not `degraded`).

Distinguishing (1) from (2) requires a raw-HTML-byte signal from the
fetch layer before extraction runs. This signal is not in today's
envelope; the provenance entry carries only extract-level status. See
the follow-up issue in §Recommendation.

Regardless of which of (1)/(2)/(3) dominates: from a partner-agent's
downstream perspective the envelope's `error.code = "empty_response"`
plus `suggestion = "site returned HTTP 200 with effectively no content;
try --ignore-robots or check with a browser"` frames this as a thin-
brochure case. A partner deciding on the error code alone would treat
these sites as having no content and skip synthesis — which the
LLM-assisted pre-review suggests would be wrong for 6/7 of the cases
in this sample (pending human confirmation).

## Recommendation

Per the issue's four-path menu, the correct call is a **split**:

1. **Keep the 1024-byte floor.** The false-positive rate *against the
   question the floor is meant to answer* is 0 % on n=30 fresh live
   sites, concordant with the archived-corpus re-classification
   showing the same bimodal distribution. This is the recommendation
   the issue explicitly permits when FP rate < 10 %.

2. **Follow-up issue opened: [#109](https://github.com/dmthepm/companyctx/issues/109).**
   Disambiguate `empty_response` from
   `extraction_failed_on_content_bearing_site`. The LLM-assisted
   pre-review flagged 6/7 `good-catch` rows as likely content-bearing
   mislabels; this rate needs human confirmation before #109 is acted
   on, but even a fraction of that rate is a separate class of problem
   the floor cannot fix — reducing the floor to 512 or 768 would have
   no effect on this sample because all 7 were 0 B, not 500-900 B.
   Candidate second heuristic: record raw-HTML length at the fetch
   layer and split the error code when
   `raw_html_bytes > N and extracted_bytes == 0`. That surfaces the
   "fetch got stripped or JS-walled" case honestly and lets the
   suggestion point at Attempt-2 escalation (smart-proxy) instead of
   implying a dead site.

The **niche-tune** path from the issue menu is not warranted: the
observed 7 failures cluster by *site template / CDN*, not by niche;
real-estate-staging and chiropractic both contain both `ok` and 0 B
cases.

## Changes to release posture

- **v0.4.0 ships as-is.** The honesty-contract fix (thin bodies stop
  laundering as `ok`) holds against fresh traffic. The floor is not
  too aggressive on the question it's meant to answer.
- **`docs/RISK-REGISTER.md` §FM-7 updated** to note the observed
  conflation between true FM-7 (site is thin) and what I'm calling
  FM-7b (site has content; extractor returned 0 B). Both currently
  surface as `error.code: "empty_response"`. Follow-up issue will
  propose splitting them.

## What this measurement does NOT cover

- **n=30 fresh sample is a probe, not a proof.** A 100-site run across
  more niches would tighten the threshold-fit claim further, but this
  sample is already enough to reject the original hypothesis (the floor
  catches legitimate 700-900 B sites at some measurable rate) — the
  rate is 0 across 30 tries on niches specifically biased to trigger
  it.
- **Does not quantify the (1)/(2)/(3) mix.** That requires the raw-HTML
  signal proposed in the follow-up issue, which is out of scope for
  COX-54's decision-only charter.
- **Does not re-validate the archived corpus under fresh fetches.**
  Sites in the COX-46 209-site sample may have changed since
  2026-04-22; this probe intentionally picks from the same trackers
  to stay in-distribution, so some overlap of hostnames with the
  archived corpus is expected and does not invalidate the fresh-live
  claim.
