# v0.1.0 release-readiness

## Decision

**Ship `v0.1.0`.** The 100-site durability run against Joel's D100 seed list
returned a **95% success rate** (envelope `status: ok` with non-empty
`homepage_text`), well above the 60% release-gate threshold. No anti-bot
blocks, no robots disallows, no HTTP errors. Failure tail is 3% network
timeouts and 2% HTML responses with no extractable body content — both
are correctly surfaced by the envelope.

## Validated on

- **Joel's D100 medical/aesthetic cohort.** 10 niches × 10 hosts = 100
  distinct URLs drawn deterministically from
  `new-signal-studio/research/*-prospect-list.md`. Details and methodology
  in [`fixtures/durability-report-2026-04-21.md`](../fixtures/durability-report-2026-04-21.md).
- **Zero-key stealth fetcher:** `curl_cffi` pinned to `impersonate="chrome146"`
  per [`decisions/2026-04-20-zero-key-stealth-strategy.md`](2026-04-20-zero-key-stealth-strategy.md).
- **Single-provider M2 orchestrator** shipped in PR #19 (commit `eda1734`).
  Attempt-2 (smart-proxy) and Attempt-3 (direct-API) are not in this
  release.
- **Real-world fetch path against 100 live sites.** `robots.txt` honored;
  no bypass flag flipped.

## Not validated on — explicit caveats

This is the critical section. The 95% success claim applies narrowly:

- **Verticals.** Seed list is 100% medical / aesthetic SMBs (plastic
  surgery, dental, orthodontics, fertility, bariatrics, hair transplant,
  immigration / divorce / lending adjacent services). **Other verticals
  (gutter / roofing / IV therapy / home services / waste management)
  have not been measured at scale.** The RISK-REGISTER taxonomy was
  mined from those verticals, and the failure shapes transfer in kind —
  but rates may not.
- **SPA-heavy shapes.** The TLS spike
  ([`research/2026-04-21-tls-impersonation-spike.md`](../research/2026-04-21-tls-impersonation-spike.md))
  already flagged that Joel's D100 lists are WordPress-dominated; the
  20-site probe surfaced zero Wix / Webflow / React/Vue SPA roots. The
  100-site run surfaced one JS-redirect-to-lander SPA pattern but no
  React/Vue/Angular single-page bundles. Performance on SPA-heavy ICPs
  is not characterised.
- **Anti-bot coverage.** Zero FM-1 403s fired in this run. That is
  encouraging evidence for `curl_cffi@chrome146` freshness, but the
  stale-fingerprint decay documented in the TLS spike will bite at some
  point — `v0.1.0` ships without Attempt-2 (smart-proxy) as a fallback.
- **Provider breadth.** Only `site_text_trafilatura` ships in `v0.1.0`.
  Review providers (`reviews_google_places`, `reviews_yelp_fusion`) and
  mention providers (`mentions_brave_stub`) are scoped for follow-up
  milestones. `partial` envelope states will only become meaningful once
  those land.
- **Longitudinal drift.** The 2026-04-21 measurement is a point-in-time
  snapshot. Fingerprint freshness, site redesigns, and CDN policy
  changes all cause drift. A rerun is due once any Attempt-2 provider
  ships.

## Release-gate thresholds

| Gate | Threshold | Measured | Pass |
|---|---|---|:-:|
| Success rate (`status: ok` + non-empty extract) | ≥ 60% | 95% | ✓ |
| Any single failure mode ≥ 5% | if so, requires a dedicated fixture | highest mode = 3% (FM-13) | ✓ (no fixture required on this trigger) |
| ≥ 2 real-world failure signatures committed as fixtures | acceptance bullet on issue #22 | 2 (`fm6-js-redirect-root`, `fm6-maintenance-page`) | ✓ |
| Register failure modes mapped | every observed mode → a register row or an explicit register-refinement note | all observed modes (FM-6, FM-13, FM-7) already in register | ✓ |

## FM-13 register handling

FM-13 fired at 3/100 = 3% in the medical/aesthetic cohort — below the 5%
refinement threshold, and without a mechanistic explanation (no shared
CDN / CMS / hosting pattern identifiable from three data points). Per
the decision rule for this PR, the register's main FM-13 body is
**unchanged**; a non-normative candidate-refinement footer is appended
to [`docs/RISK-REGISTER.md`](../docs/RISK-REGISTER.md) noting the
observation and calling for vertical-diverse replication before any
material rewording.

## Follow-ups (separate issues, not this release)

- **Mock-block provider hook.** `site_text_trafilatura` cannot currently
  simulate network-level failures in `--mock` mode. Adding a fixture-side
  sentinel (e.g. `fixture-block.txt`) unblocks FM-13 / FM-1 / FM-5
  regression fixtures. One-PR follow-up; pointer back to this durability
  run as the motivating evidence.
- **Vertical-diverse durability rerun.** Rerun this methodology on a
  gutter / roofing / waste-management cohort (the verticals the
  RISK-REGISTER transcripts came from). If FM-13 rates converge there,
  the register main body can be refined with confidence; if they diverge,
  the medical/aesthetic-only caveat stays in place.
- **SPA-root probe.** A deliberate Wix / Webflow / React-SPA probe is
  already tracked as
  [issue #32](https://github.com/dmthepm/companyctx/issues/32); the
  100-site run does not close it (one JS-redirect hit is not
  SPA-characterising coverage).
- **Default-timeout tuning.** Provider default is 10s; p90 of ok-latency
  is 7.7s. A handful of sites time out at the 10s boundary. Shipping
  `v0.1.0` at the current default is fine; a follow-up could measure
  sensitivity.

## Rollback / "how to know this was wrong"

Evidence that would retract this decision, in order of severity:

1. A real-world user run surfaces `status: ok` on an anti-bot 403 — means
   the envelope is lying; immediate hotfix.
2. A vertical-diverse rerun measures < 60% on any ICP Joel's pipeline
   actually uses — means the 60% gate was niche-overfit.
3. Fingerprint decay on `chrome146` drops the measured rate below 80%
   within a quarter — means Attempt-2 becomes a release-blocker for
   `v0.1.1`.

None of these is observable today. Ship.
