# v0.1.0 release-readiness

## Decision

**Ship `v0.1.0`.** The 100-site durability run against Joel's D100 seed
list returned **97 / 100 envelope `status: ok`** and **zero
anti-bot-blocked / robots-blocked / HTTP-error responses**. That
comfortably clears the 60% `status: ok` release-gate on issue #22.

Every number in this ADR is traceable to
[`research/2026-04-21-durability-sample-raw.jsonl`](../research/2026-04-21-durability-sample-raw.jsonl)
(slug-only, 100 rows) and to
[`classify()` in `scripts/run-durability-batch.py`](../scripts/run-durability-batch.py).
No hand-edited numbers.

## Measured rates

| Outcome | Count | Rate |
|---|---:|---:|

| `status: ok`, extracted `homepage_text` < 1 KB (FM-7) | 21 | 21% |

## Validated on

- **Joel's D100 medical/aesthetic cohort.** 10 niches × 10 hosts = 100
  distinct URLs drawn deterministically from
  `new-signal-studio/research/*-prospect-list.md`. Details in
  [`fixtures/durability-report-2026-04-21.md`](../fixtures/durability-report-2026-04-21.md).
- **Zero-key stealth fetcher:** `curl_cffi` pinned to
  `impersonate="chrome146"` per
  [`decisions/2026-04-20-zero-key-stealth-strategy.md`](2026-04-20-zero-key-stealth-strategy.md).
- **Single-provider M2 orchestrator** shipped in PR #19.
- **Real-world fetch against 100 live sites.** `robots.txt` honored
  throughout; no bypass flag flipped.

## Not validated on — explicit caveats

- **Verticals.** Seed list is 100% medical / aesthetic SMBs. Other
  verticals (gutter / roofing / IV therapy / home services / waste
  management) are untested at scale. The RISK-REGISTER taxonomy transfers
  in kind, but frequencies may not.
- **SPA-heavy shapes.** The 20-site TLS spike already flagged that Joel's
  D100 list is WordPress-dominated. The 100-site run surfaced one
  JS-redirect-to-lander shape but no React/Vue/Angular single-page
  bundles. Performance on SPA-heavy ICPs is not characterised. Tracked
  as [issue #32](https://github.com/dmthepm/companyctx/issues/32).
- **Anti-bot coverage.** Zero FM-1 403s in this run. That's encouraging
  evidence for `curl_cffi@chrome146` freshness, but stale-fingerprint
  decay is real (documented in the TLS spike). `v0.1.0` ships without
  Attempt-2 (smart-proxy) as a fallback.
- **Provider breadth.** Only `site_text_trafilatura` ships in `v0.1.0`.
  `partial` envelope states become meaningful once review / mention
  providers land.
- **Structured-signal extraction.** The register's FM-6 ("raw text
  captured, structured fields missed") cannot fire in M2 because
  `signals_site_heuristic` isn't wired yet. A rerun once that provider
  ships will yield the real FM-6 rate.
- **Longitudinal drift.** The 2026-04-21 measurement is a point-in-time
  snapshot. Fingerprint freshness, site redesigns, and CDN policy
  changes all cause drift. A rerun is due once any Attempt-2 provider
  ships.

## Release-gate thresholds

| Gate | Threshold | Measured | Pass |
|---|---|---|:-:|
| `status: ok` rate | ≥ 60% | 97% | ✓ |
| Any single failure mode ≥ 5% | if so, needs a dedicated fixture | highest = 21% (FM-7) | ✓ (fixtures promoted below) |
| ≥ 2 real-world failure signatures committed as fixtures | acceptance bullet on #22 | 2 (`fm7-js-redirect-root`, `fm7-maintenance-page`) | ✓ |
| Register failure modes mapped cleanly | every observed `fm_code` maps 1:1 | FM-7 (21%), FM-13 (3%); FM-6 0% honest | ✓ |

FM-7 crosses the 5% threshold — and the two fixtures promoted this PR
are FM-7 exemplars, satisfying the threshold's "dedicated fixture"
requirement.

## FM-13 register handling

FM-13 fired at 3/100 = 3% — below the 5% refinement threshold, no
shared mechanism identifiable from three data points. Per the decision
rule for this PR, the register's main FM-13 body is **unchanged**; a
non-normative candidate-refinement footer was appended to
[`docs/RISK-REGISTER.md`](../docs/RISK-REGISTER.md) noting the
observation and calling for vertical-diverse replication before any
material rewording.

One of the three FM-13 rows (`fertility-23`) recorded a ~20 s provider
latency against a 10 s configured timeout. A clean re-probe of the
same host returns at 10 s as expected; the 20 s is a transient
artifact (likely slow DNS resolution before libcurl's timer starts or
resolver retries), not an `/about` follow-up (impossible per the
provider code path) and not a reproducible provider anomaly.

## Follow-ups (separate issues)

- **Mock-block provider hook** (#40) — lets future FM-13 / FM-1 / FM-5
  fixtures be committed as regression guards.
- **Vertical-diverse durability rerun** — gutter / roofing /
  waste-management cohort; unblocks a confident register refinement if
  FM-13 rates converge there.
- **SPA-root probe** (#32) — deliberate Wix / Webflow / React-SPA
  probe.
- **Default-timeout tuning** — provider default is 10 s; p90 of ok
  latency is 7.7 s. A sensitivity measurement is worth its own issue.

## Rollback / "how to know this was wrong"

Evidence that would retract this decision, in severity order:

1. A real-world user run surfaces `status: ok` on an anti-bot 403 —
   envelope is lying; immediate hotfix.
2. A vertical-diverse rerun measures < 60% `status: ok` on any ICP
   Joel's pipeline actually uses — 60% gate was niche-overfit.
3. Fingerprint decay on `chrome146` drops measured rate below 80%
   within a quarter — Attempt-2 becomes a release-blocker for
   `v0.1.1`.

None of these is observable today. Ship.
